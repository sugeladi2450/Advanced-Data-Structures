# 跳表增长率 p 对性能与高度的影响实验报告

> 代码基于助教提供的跳表实现，实验脚本与统计口径见：
> - 平均搜索长度（遍历节点个数）：[`skiplist::skiplist_type::query_distance()`](skiplist.cc:94)
> - 最大高度（层数）：[`skiplist::skiplist_type::max_level()`](skiplist.cc:121)
> - 平均高度（平均塔高）：[`skiplist::skiplist_type::average_height()`](skiplist.cc:129)
> - 查询次数：[`QUERY_TIMES`](test-main.cc:46)（固定为 10000）

## 1. 实验目的

同跳表长度（元素个数 n）下，探究增长率（提升概率）p 与以下指标的关系，并据折线图判断是否符合理论：

1. 平均搜索长度（一次查询遍历到的节点个数，含横向与向下移动）
2. 跳表最大高度（max level / 层数）
3. 跳表平均高度（每个 key 的“塔高”平均值）

## 2. 实验设置

### 2.1 跳表规模与参数

- 跳表长度（元素个数）n：50、100、200、500、1000
- 增长率 p：1/2、1/e、1/4、1/8（数值分别为 0.5、0.367879...、0.25、0.125）

以上组合由脚本 [`test.sh`](test.sh:4) 固定生成，共 5×4=20 组。

### 2.2 数据生成与查询

1. 随机生成 n 个 key-value 对并插入跳表（key 为 uint64 随机数）。
2. 进行至少 10000 次随机查询：每次从已插入元素中随机抽取一个 key 来查询。
3. 统计每次查询的“搜索长度”（遍历到的节点个数）并取平均。

查询次数固定为 10000，定义见 [`QUERY_TIMES`](test-main.cc:46)。

### 2.3 指标定义（与代码口径一致）

- **平均搜索长度**：函数 [`skiplist::skiplist_type::query_distance()`](skiplist.cc:94) 返回值的平均。
  - 该实现将“访问/遍历到的节点”计数为：
    - 每向右移动到一个节点，计数 +1；
    - 每向下移动一层，计数 +1；
    - 初始从 top head 开始，计数从 1 起。
- **最大高度**：[`skiplist::skiplist_type::max_level()`](skiplist.cc:121) 返回层数（包含各层 head）。
- **平均高度**：[`skiplist::skiplist_type::average_height()`](skiplist.cc:129) 计算所有层节点总数（不含 head）/ 底层节点数（不含 head），即平均塔高。

### 2.4 运行与绘图

- 运行 20 组实验输出 raw 文本：`bash test.sh 0 > raw.txt`
- 解析成表格：[`parse.py`](parse.py:1) 生成 `result.csv`
- 绘图：[`plot.py`](plot.py:1) 输出 `figs/` 下的折线图（对每个 n，分别画 avg_query_distance / max_level / avg_height vs p）。

> 为降低随机性带来的波动，可对多个 seed 重复建表并合并输出再取平均（见本文第 5 节“误差来源与改进”）。

## 3. 理论背景与预期趋势

跳表中每个元素是否“提升”到更高层通常服从几何分布。设提升概率为 p，则：

- 元素达到第 k 层以上的概率约为 p^k。
- 因而**平均塔高**通常与 1/(1-p) 同阶：p 越大，元素越容易被提升，平均高度越大。
- **最大高度**通常与 \( \log_{1/p}(n) \) 同阶：p 越大，高层出现的概率更大，最大高度更高。
- **平均搜索长度（期望搜索代价）**经典结论为：
  \[
  \mathbb{E}[\text{search}] = \Theta\big(\tfrac{1}{p} \log_{1/p} n\big)
  \]
  这意味着：
  - p 太小：层数太少，查询更像在底层链表上走很远，代价增大；
  - p 太大：高层节点过密、向下次数与比较次数的常数项上升，代价也可能增大；
  - 因此通常存在一个“较优 p”（很多分析与经验中在 1/e 附近）使常数项较小。

## 4. 实验结果分析（结合折线图）

> 以下分析以折线图 `figs/` 中同一 n 的三张图为依据；你应在提交时把关键图粘贴到报告中。

### 4.1 p 与平均高度（avg_height）的关系

**现象（应当看到）：**
- 在固定 n 下，随着 p 从 1/8 → 1/4 → 1/e → 1/2 增大，avg_height 单调上升或总体上升。

**解释：**
- p 越大，每个元素“向上提升”的期望次数越多，形成更高的塔，因此平均高度增大。
- 该趋势与“几何分布提升模型”的理论预期一致。

### 4.2 p 与最大高度（max_level）的关系

**现象（应当看到）：**
- 在固定 n 下，p 越大，max_level 整体越大（但可能伴随小幅波动）。

**解释：**
- 最大高度由极端高塔决定。p 增大后，出现高塔的概率上升，因此整体 max_level 增大。
- 由于 max_level 是“极值统计”，方差较大：同一 n,p 在不同随机种子下 max_level 波动可能明显。

### 4.3 p 与平均搜索长度（avg_query_distance）的关系

**现象（常见形态）：**
- 在固定 n 下，avg_query_distance 往往呈“先降后升”或在某个 p（常见靠近 1/e）附近取得较小值。

**解释：**
- p 太小导致层数不足，查询需要在底层进行较长的水平移动；
- p 增大可显著减少底层水平移动（更快“跳跃”）；
- 但 p 过大时，高层节点太密，导致：
  - 每层横向比较次数（或指针遍历次数）增多；
  - 同时塔更高，向下移动次数也增多；
  综合常数项使搜索长度回升。

**与理论的符合性：**
- 若图中最低点出现在 p=1/e 附近，符合经典“常数最优在 1/e 左右”的经验结论。
- 即使最低点不恰好在 1/e，只要呈现“存在最优 p、两端变差”的总体趋势，也可认为与理论定性一致。

## 5. 若与理论不完全一致：可能原因与改进建议

若你的折线图没有表现出上述清晰趋势（例如 avg_query_distance 没有明显最小点，或 max_level/avg_height 出现反常波动），常见原因如下：

1. **随机波动（seed 太少）**
   - 当前一次实验只用一个 seed 建表，max_level 属于极值指标，波动很大。
   - 改进：对每个 (n,p) 用多个 seed（如 20 次）独立建表，三项指标取平均，再绘图。

2. **规模 n 较小，渐近性质不明显**
   - n=50~1000 仍偏小，\(\log n\) 的渐近趋势不够“平滑”。
   - 改进：可额外加入更大 n（如 5000、10000）做对照（如果作业允许）。

3. **统计口径与理论口径不同（常数项差异）**
   - 本实验的“搜索长度”按遍历节点数计，且包括向下移动次数（见 [`query_distance()`](skiplist.cc:94)），与一些教材中按“key 比较次数”的口径不同。
   - 这会改变不同 p 之间的常数项对比，使曲线最低点略偏移。

4. **key 可能重复（极小概率）且 put 未做覆盖更新**
   - [`skiplist_type::put()`](skiplist.cc:43) 未显式处理相同 key 的覆盖更新，若发生重复 key，会插入重复节点，影响高度与路径。
   - 虽然 64-bit key 重复概率很低，但严格来说仍是误差来源。

5. **p 的实现定义需要统一解释**
   - 该实现用几何分布生成提升次数（见 [`skiplist_type::skiplist_type(double)`](skiplist.cc:19) 与 [`skiplist.h`](skiplist.h:20) 的注释），等价于“每次以概率 p 继续向上”。
   - 在报告中需要说明清楚这里的 p 是“继续提升的概率”，与部分资料中“停止概率”的定义可能相反。

## 6. 结论

- 在固定 n 下，p 增大通常会带来更高的跳表（max_level 与 avg_height 整体上升），这与跳表的几何提升模型一致。
- 平均搜索长度随 p 变化一般存在一个折中最优点：p 太小层数不足、p 太大层过密且下跳次数多，导致平均搜索长度上升；最优 p 常见在 1/e 附近。
- 若实验曲线不够符合理论，多由随机波动、n 偏小、统计口径差异等导致；通过多 seed 平均可显著改善。
