“在本项目中，我将 LLM 作为一个带有记忆和推理能力的Agent来使用。具体过程如下：
通过SKILL.md，我用自然语言向 LLM 描述了 BFS 的算法思想。赋予 LLM 调用外部数据源的能力（如 neighbors 和 kv_get 工具）。LLM 根据当前看到的图结构，自主决定下一步该查哪个节点，输出 JSON 格式的工具调用请求；Python 脚本拦截该请求并查询本地数据库，将真实结果返回给 LLM。强制要求 LLM 在每一步调用前，在文本中写下 [状态记录] visited = [...]，利用上下文窗口充当 BFS 的“内存队列”，直到触发终止条件并输出最终答案。
优点：
灵活性高与泛化能力强： 不需要硬编码复杂的数据结构（如 Python 的 Queue 或 Set）。只需修改几句自然语言 Prompt，LLM 就能立刻从“图遍历模式”切换到“KV 存储并行查询模式”，甚至能直接理解“按升序排列”、“过滤空值”等复杂指令。
过程可解释性强： 传统的死循环很难调试，但因为我们强制 LLM 输出了思维链（如控制台打印的 [状态记录]），我们可以像读心术一样，清晰地看到 LLM 每一步是怎么思考的，为什么它决定去查节点 B 而不是节点 C。
局限性：
极高的延迟与时间开销：传统 BFS 遍历几个节点只需要几微秒，而交由 LLM 调度时，每推进一步都需要经历一次完整的 HTTP 网络请求。
上下文脆弱与幻觉 ：LLM 本质上是基于概率的文本接龙模型。当层数变多、历史记录变长时，LLM 会出现严重的上下文遗忘或格式崩溃。例如，它会在纯文本中“假装”调用工具，或者错误理解 Python 的 None 返回值从而输出全集。

3. **Tool Calling 与 Prompt 工程**
   - 对比 Tool Calling API 和自由文本 JSON 输出，你觉得各有什么优劣？
```
原生 Tool Calling API：
优势： 稳定性极高。底层模型经过专门的微调，能强保证输出符合 JSON Schema 规范的参数；API 结构清晰，代码层面直接通过 msg.tool_calls 获取，无需复杂的字符串解析。
劣势： 依赖特定模型（如 gpt-4o 完美支持，但部分纯推理模型如原生 deepseek-r1 可能不支持）。
自由文本 JSON 输出：
优势： 通用性强。任何一个能对话的基础模型都能使用。
劣势： 极易出错。模型常常会加上 Markdown 标记（```json）、多余的闲聊文字，甚至输出残缺的 JSON，导致必须在代码中写极其复杂的正则表达式去“兜底”解析。
```
   - 你在设计 prompt 和 function schema 时遇到了哪些挑战？做了哪些调整？
```
挑战 1：死循环与遗忘。 模型在遍历几层后忘记了已访问节点。
调整： 在 SKILL.md 中加入强制状态记录规则，要求它在调用工具前必须显式打印 [状态记录] visited = [...]，充当外挂内存。
挑战 2：幻觉假调用。 模型在纯文本里写“正在调用工具”，但没触发真正的 API。
调整： 在 Prompt 中加入严厉的“违者报错”警告，禁止角色扮演。
挑战 3：空值引发的全集输出。 KV 查询时，不存在的 key 返回 None，被模型误认为有效字符串。
调整： 在 Python 运行时层面做“防呆翻译”，将 None 强行转换为大模型绝对不会误解的 NOT_FOUND。
```
4. **规模扩展与系统设计**
   - 假设图的节点数扩展到 \(10^5\) 或更大，你觉得本 Lab 中这种"把局部结构编码进 prompt 再让 LLM 算"的方式是否还能正常工作？请从以下角度进行分析：
     - prompt 长度与模型上下文限制；
     - 调用次数与时间开销；
     - 错误率和可验证性。
```
无法正常工作。
prompt 长度与模型上下文限制:LLM 的上下文窗口有限.当 visited 列表增长到几万个节点时，不仅会撑爆上下文，还会触发“Lost in the Middle”效应，导致模型完全丧失推理能力。
调用次数与时间开销： BFS 遍历10^5个节点，在我们的框架下需要发起数万次 HTTP 请求。即使网络完美，每次 API 耗时 1 秒，也要花几十个小时，这在工程上是不可接受的。
错误率和可验证性:只要在这上万次循环中，模型出现一次 JSON 格式错误或幻觉，整个算法结果就会全盘崩溃，且极难 Debug。

大模型负责“意图解析”： 用户输入自然语言（“帮我找距离 A 最远的节点”），LLM 将其解析为确定的传统数据库查询语言或算法指令。
传统算法负责“图遍历 / KV 查询” ：拿到 LLM 解析出的指令后，后端的Python脚本或图数据库引擎使用传统的高效算法去执行，得到最终结果。
大模型负责“总结” ：将传统算法算出的最终结果喂给 LLM，让 LLM 用人类友好的自然语言回答用户。
如果 LLM 无法解析用户的查询，或者生成的查询语句执行报错，系统应捕获异常，并回退到经典的 GUI 搜索框，或者返回固定的兜底提示：“我目前无法理解该复杂查询，请提供明确的节点 ID”。
