# Lab1 报告

## 一、实现与设计

### 1. 图操作的实现思路

本 Lab 的写操作都围绕三个结构同步维护：节点属性表 `_nodes`、出边邻接表 `_adj`、入边反向索引 `_radj`。

插入节点时，如果节点不存在就初始化属性、出边表和入边集合；如果节点已存在且传入了属性，就合并覆盖属性。插入边时先确保两个端点都存在，再更新 `_adj[from_id][to_id] = weight`，同时把 `from_id` 加入 `_radj[to_id]`。删除边时同时从 `_adj` 和 `_radj` 中清掉对应关系。删除节点时要删除它的所有出边和入边，最后再删除节点本身。

最容易写错的是只维护出边表而忘记维护反向索引，或者删除节点时只删出边、不删其他节点指向它的入边。为避免这种问题，我把真正修改内存结构的逻辑集中到 `_apply_*` 方法里，WAL 恢复和正常写入都复用同一套内存更新逻辑。

`delete_node` 必须处理入边，因为其他节点的出边表中可能仍然指向被删除节点。如果不清理，这些边会变成悬挂边，后续依赖查询会返回已经不存在的节点。没有反向索引时，只能遍历全部节点的邻接表寻找入边，复杂度接近 O(V + E)；维护 `_radj` 后，只需要访问被删节点的入边集合和出边集合。

### 2. WAL + Snapshot 的故障模型与设计空间

写入时先把操作记录追加到 WAL，再更新内存结构。恢复时先加载 snapshot 得到一个较新的基础状态，再逐行重放 WAL，把 snapshot 之后发生的写操作补回来。这样即使程序在写入后、关闭前中断，也可以通过 WAL 恢复到最新状态。

Snapshot 的作用是给恢复过程提供一个压缩后的基线。如果只有 WAL，日志会不断增长，恢复时需要从最早的操作开始重放，启动时间会越来越长，磁盘空间也会持续膨胀。

真实系统中可以优化的地方包括：根据 WAL 大小、时间间隔或系统负载动态触发快照；使用后台线程做增量或异步 snapshot，减少前台写入阻塞；snapshot 写入时使用临时文件和原子替换，避免崩溃时留下半写入文件；还可以在 snapshot 后截断 WAL，控制恢复时间和磁盘占用。

### 3. 图算法的语义与复杂度边界

`get_all_dependencies` 适合回答“这个包最终会依赖哪些包”这类完整影响面问题。例如排查某个业务包是否间接依赖漏洞库时，可以查询该业务包的传递闭包，再判断漏洞库是否在结果中。

`get_dependencies_by_layer` 适合回答“依赖是通过几层传递过来的”这类需要分层解释的问题。例如某个核心包第 1 层不依赖漏洞库，但第 2 层依赖，说明风险来自间接依赖链。

引入 `max_layers` 是为了限制查询开销。依赖图可能很大，如果无限制地展开，结果数量和运行时间都可能膨胀。`max_layers` 牺牲了一部分完整性，换来可控的查询成本。如果用户需要继续深挖，可以先查询前几层定位方向，再把感兴趣的边界节点作为新的起点继续分层查询。

## 二、Skill 与 Agent

### 4. 算法归属问题的本质

本 Lab 把图算法放在后端，LLM 负责调度和汇总，这是因为图计算需要确定性、可复现和较高效率。最短路径、传递闭包、按层 BFS 都适合由程序精确执行，而不是交给 LLM 在文本中猜测。

这种分工的好处是结果更可靠，复杂查询可以复用后端算法，LLM 只需要决定调用哪个工具、用什么参数、如何根据返回结果组织答案。代价是系统需要提前设计好工具接口；如果接口缺少某种查询能力，LLM 只能绕着现有工具多轮调用。

在 Phase 4 的漏洞依赖调查中，LLM 的主要角色是理解任务、对每个核心包调用依赖查询工具、比较返回结果中是否包含漏洞库，最后按要求输出包名列表。

### 5. 渐进式披露的边界

description 中最关键的信息包括：工具能回答什么问题、输入参数名和含义、返回值格式、无结果时返回什么、层数或路径语义是什么。比如 `get_dependencies_by_layer` 必须说明第 1 层是直接依赖，`max_layers` 是最大展开层数。

不适合写太多实现细节，例如内部使用 `_adj`、`_radj`、WAL 或 snapshot 的细节。这些信息不会帮助 LLM 正确调用工具，反而可能干扰它关注任务本身。description 应该描述接口语义，而不是后端实现。

### 6. 接口的完备性与最小性

现有四个工具对基本的软件包依赖分析和漏洞影响分析已经够用：可以查直接依赖、最短路径、全部依赖和分层依赖。它们覆盖了“是否依赖”“怎么依赖”“依赖多深”这些核心问题。

最容易缺的是反向影响分析能力：给定一个漏洞库，直接找出哪些包依赖它。现在需要对候选核心包逐个调用 `get_all_dependencies` 再筛选。如果只能新增一个工具，我会添加 `get_reverse_dependencies(node, max_layers)`，输入漏洞包和最大反向层数，输出按层依赖它的上游包，例如 `{1: [...], 2: [...]}`。这样可以显著减少漏洞影响分析中的调用轮数，也能降低 LLM 漏查候选包的概率。
