## Lab1 报告与思考题

### 一、实现与设计

1. **图操作的实现思路（不需要重复描述你用了什么结构）**
   - 请你大概描述一下本Lab中图操作的实现思路。请概括：插入/删除点、插入/删除边这些写操作，你分别做了哪些关键更新步骤？哪里最容易写错或漏掉、你是如何避免的？
   - `delete_node` 为什么需要处理入边？如果没有反向索引，你会怎么做、会带来什么后果？

2. **WAL + Snapshot 的故障模型与设计空间**
   - 你如何用 WAL 和 Snapshot 保证「断电重启后图仍然一致」？请用流程描述：写入时做了什么、恢复时按什么顺序做了什么。
   - 为什么需要 Snapshot？如果只有 WAL、日志越来越长，会出现什么现实问题？
   - 在真实系统里，“何时做快照/如何做快照”有哪些可优化空间？（写你觉得最关键的2-3点即可。可以从这些角度自由展开：快照触发条件如何设定、如何降低恢复时间、如何控制磁盘/IO 开销、崩溃在快照进行到一半时如何保证可恢复性等。）

3. **图算法的语义与复杂度边界**
   - `get_all_dependencies` 和 `get_dependencies_by_layer` 分别适合回答什么问题？请结合依赖分析/漏洞排查的场景举例说明。
   - 为什么要引入 `max_layers`？它在“结果是否完整”和“查询是否可控”之间做了什么取舍？如果用户想继续往更深层查，你会如何组织一次或多次查询（自由发挥即可）。

### 二、Skill 与 Agent

4. **算法归属问题的本质**
   - 本 Lab 将图算法放在后端、LLM 负责调度与汇总。请结合 Phase 4 的体验解释：为什么不把“图的计算”直接交给 LLM？这种分工带来了哪些好处、又牺牲了什么？
   - LLM 在 Skills 使用中主要扮演了哪些角色（例如决定调用顺序、选择工具、组织最终回答等）？可结合具体任务说明（可以不是Lab中任务）。

5. **渐进式披露的边界**
   - Skill 的“按需披露”意味着 LLM 只看到工具签名与描述，而看不到实现。你认为哪些信息写在 description 里最关键，能显著减少误用/误解？哪些信息反而写了也没用、甚至会干扰？

6. **接口的完备性与最小性**
   - 现有四个工具对“软件包依赖分析”、“漏洞包影响分析”等任务来说够用吗？你觉得最容易缺的是哪一类能力，为什么？
   - 如果让你新增 **1 个**工具（只允许 1 个），你会加什么？它的输入输出大概长什么样，能显著减少调用轮数或降低出错概率吗？
