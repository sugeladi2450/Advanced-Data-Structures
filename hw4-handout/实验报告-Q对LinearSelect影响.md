# Linear Select 算法中分组大小 Q 的影响探究

## 一、实验目的

在经典的 Linear Select（Median of Medians）算法中，数据被划分为大小为 $Q$ 的子组，取每组中位数后再递归求出“中位数的中位数”作为划分的 Pivot。经典理论分析中通常取 $Q=5$ 以保证最坏情况下的 $O(n)$ 复杂度。但在现代计算机体系结构下，$Q$ 的选择不仅影响算法的理论递归深度，更直接关系到缓存命中率、分支预测和内存分配等工程性能指标。

本实验旨在系统评估参数 $Q$ 对算法实际运行效率的综合影响，具体探究：

1. $Q$ 的取值对执行时间、递归深度及调用总次数的影响。
2. 极端小 $Q$ 值（如 $Q=3$）的性能退化根源。
3. 不同数据分布（随机、顺序、重复、近似有序）下 $Q$ 的适应性差异。

---

## 二、实验设计与环境

### 2.1 算法实现与控制变量

为确保测试反映算法本身的特性而非底层实现的额外开销，实验在工程层面做了如下严格控制：

* **内存分配机制**：算法全程采用 **In-place（原地交换）** 策略，避免了每层递归中动态分配 `std::vector` 带来的 $O(n)$ 级别内存申请开销。
* **组内中位数求法（Base Case）**：为公平对比不同大小的 $Q$，当 $Q \le 15$ 时采用插入排序（或局部最优排序网络）；当 $Q > 15$ 时，调用标准库 `std::nth_element` 获取组内中位数。
* **对照组设置**：引入基于随机 Pivot 的尾递归 QuickSelect 作为基准基线。两算法共享相同的内部三路划分（Three-way Partition）逻辑。

### 2.2 测试参数
* **$Q$ 取值**：3, 5, 7, 9, 11, 15, 21, 31, 41, 51, 75, 101
* **数据规模 $N$**：100 ~ 20000 级（核心基准测试取 $N=5000$）
* **数据分布**：完全随机、严格顺序、含 10% 独立值的重复序列、近似有序（少量逆序对）
* **环境**：x86-64 架构，g++ (C++20) `-O2` 编译。每次测试随机生成目标排名 `nth`，循环 200~500 次取统计值。

**直接观察：**

1. **$Q$ 与性能高度正相关**：随着 $Q$ 的增大，执行时间显著下降。$Q=101$ 时的 P90 耗时仅为 $Q=3$ 时的 1/6。
2. **递归调用次数锐减**：$Q$ 从 3 增至 101，递归深度仅下降约 40%（12.1 降至 7.0），但**总递归调用次数下降了 98%**（766 次暴跌至 13 次）。
3. **与随机基准的对比**：即使在最优的 $Q=101$ 下，Linear Select（68μs）依然不敌纯随机的 QuickSelect（36μs），由于寻找安全 Pivot 的确定性常数开销始终存在。

### 3.2 数据分布的敏感度分析

测试 $Q=5$ 与 $Q=21$ 在不同分布下的 P90 耗时（μs）：

| 数据分布 | Linear Q=5 | Linear Q=21 | QuickSelect |
| --- | --- | --- | --- |
| 随机 | 130 | 102 | 39 |
| 重复 | 125 | 99 | 39 |
| 近似有序 | 68 | 53 | 16 |
| 顺序 | 55 | 37 | 12 |

* **顺序/近似有序数据极速通过**：所有算法在顺序数据上均表现出 2~3 倍的性能提升。这并非由于 Pivot 质量改变（随机数据下的 Pivot 期望分割点也是居中的），而是因为顺序数据的比较操作使得底层 CPU 的**分支预测**几乎 100% 命中，流水线极少排空；同时连续读取带来了极佳的**缓存局部性**。
* **大 $Q$ 值的普适性**：无论底层数据分布如何恶劣，$Q=21$ 始终稳定优于 $Q=5$。

## 四、深入分析

### 4.1 为什么大 Q 在工程实践中跑得更快？

在经典分析中，大 $Q$ 会导致在组内寻找中位数的常数开销增大。但在现代工程环境下，这种推演存在局限：

1. **递归栈与函数调用开销远大于连续内存排序**：每次递归调用涉及状态保存、栈帧分配以及参数传递。$Q$ 的增大虽然增加了寻找局部位数的工作量，但极大压缩了中位数数组 $M$ 的规模（大小为 $n/Q$），从而把极为昂贵的**递归调用次数**（见 3.1 节数据）转嫁为了廉价的**连续内存比较操作**。
2. **内存访问模式**：在 L1/L2 缓存大小充足的情况下，对几十个元素执行一次 `std::nth_element` 操作完全在高速缓存内完成，硬件层面临界成本极低。

### 4.2 Q=3 性能灾难的原因

实验发现 $Q=3$ 耗时异常庞大，常有误解将其归结为“算法退化为 $O(n \log n)$”。这里需要严格区分**最坏情况**与**平均情况**：

**最坏情况（Worst-Case）：**
理论上，当 $Q=3$ 时，极端对抗数据确实能让算法退化为 $O(n \log n)$。中位数数组大小为 $n/3$，Pivot 只能保证排除不到 $1/3$ 的元素，递归式变为 $T(n) \le T(n/3) + T(2n/3) + O(n)$，由于子问题规模比例之和 $1/3 + 2/3 = 1$，常数无法收敛。

**实验环境的平均情况（Average-Case）：**
本实验 3.1 节采用的是**随机分布数据**。在随机数据下，$Q=3$ 选出的 Pivot 期望位置仍然非常接近整体中位数，期望划分比例约为 1:1。
其实际的期望时间复杂度递推式为：

$$E[T(n)] = E[T(n/3)] + E[T(n/2)] + O(n)$$

由于 $1/3 + 1/2 = 5/6 < 1$，**在随机数据下，$Q=3$ 的理论期望时间复杂度依然是严格的 $O(n)$。**

**真正导致其实际耗时极长的原因是：常数因子爆炸。** $5/6$ 非常接近 $1$，导致递归树每一层的工作量衰减极慢。算法必须构建庞大的递归树（测试显示达 766 次函数调用）才能将数据缩减至 Base Case，巨大的常数因子彻底拖垮了性能。

# QuickSelect vs LinearSelect 性能对比分析

## 一、实验目的

比较两种经典选择算法的实际性能：

- **QuickSelect**：随机选择 pivot，期望 $O(n)$ 时间复杂度，常数小
- **LinearSelect**（Median of Medians）：通过分组中位数保证 pivot 质量，严格 $O(n)$ 最坏情况，含可调参数 Q

探究不同数据规模、数据分布对两者的影响差异。

---

## 四、数据分布对两者的影响

### 4.1 LinearSelect Q=5 下各分布表现（N=5000）

> QuickSelect 不含参数 Q，此处作为全局基准对照组（Baseline）列出，其数据与 Q 无关。

| 数据分布 | Linear P90(μs) | Quick P90(μs) | 比值 (L/Q) |
|:-------:|:-------------:|:-------------:|:---------:|
| 随机 | 134 | 42 | 3.2× |
| 顺序 | 98 | 16 | 6.1× |
| 重复 | 132 | 43 | 3.1× |
| 近似有序 | 75 | 18 | 4.2× |

### 4.2 LinearSelect Q=21 下各分布表现（N=5000）

| 数据分布 | Linear P90(μs) | Quick P90(μs) | 比值 (L/Q) |
|:-------:|:-------------:|:-------------:|:---------:|
| 随机 | 114 | 43 | 2.7× |
| 顺序 | 26 | 12 | 2.2× |
| 重复 | 114 | 42 | 2.7× |
| 近似有序 | 55 | 16 | 3.4× |

### 4.3 分布影响分析

**各分布下的波动幅度（max/min）：**

| 算法 | 4.1 节数据 | 4.2 节数据 |
|:----|:--------:|:--------:|
| QuickSelect（基准） | 43/16 = **2.69×** | 43/12 = **3.58×** |
| LinearSelect | 134/75 = **1.79×** | 114/26 = **4.38×** |

**关键发现：**

1. **顺序数据最快**：原因并非 pivot 质量更高——随机数据下 pivot 期望上同样分割均匀。真正原因是 **CPU 分支预测（Branch Prediction）**对有序数据的比较结果几乎 100% 命中，避免流水线停顿；同时**缓存局部性（Cache Locality）**使 partition 中的内存访问完全连续，大幅减少 cache miss
2. **Q=5 时 LinearSelect 波动更小**（2.69× vs 1.79×），**Q=21 时 LinearSelect 波动更大**（3.58× vs 4.38×）——大 Q 下中位数数组变小，pivot 质量对数据分布更敏感
3. **重复数据对两者耗时与随机数据接近**：三路划分将等值元素排除在递归之外，有效缓解了重复值的不利影响

### 5.3 与 QuickSelect 的递归差异

| | QuickSelect | LinearSelect |
|:--|:----------|:------------|
| 递归方式 | 单分支（尾递归） | 双分支（中位数递归 + 选择递归） |
| 期望深度 | $O(\log n)$ | $O(\log n)$ |
| 调用次数 | 期望 $O(\log n)$ | $O(\log_Q n + \log n)$ |
| 最坏深度 | $O(n)$（退化） | $O(\log n)$（Q≥5 时保证；Q=3 退化为 $O(\log n)$） |

QuickSelect 虽然最坏情况可能退化到 $O(n^2)$，但在随机数据下几乎不会发生（概率极低）。LinearSelect 用额外的递归分支换取了确定性保证——但需注意 **Q=3 时此保证失效**，Median of Medians 在 Q=3 时最坏时间复杂度退化为 $O(n \log n)$，因为中位数数组中位数无法保证常数比例的剔除（见报告 1 第 4.4 节）。

## 七、结论

### 7.1 QuickSelect vs LinearSelect

| 维度 | QuickSelect | LinearSelect |
|:----|:-----------|:------------|
| **实际速度** |  快 1.7~6.1× |  较慢 |
| **理论保证** | 期望 $O(n)$ | Q≥5 时严格 $O(n)$；Q=3 退化为 $O(n \log n)$ |
| **最坏情况** | $O(n^2)$（罕见） | $O(n)$ |
| **分布敏感性** | 波动 2.7~3.6× | 波动 1.8~4.4× |
| **实现复杂度** | 简单 | 较复杂（分组+双递归） |

### 7.2 建议

1. **实际应用中优先使用 QuickSelect**：随机 pivot 在实际数据上几乎不会退化，且常数小得多
2. **关键任务场景选 LinearSelect（Q≥5）**：在实时系统、安全攸关系统或对最坏运行时间有严格约束的场景下，QuickSelect 的 $O(n^2)$ 退化风险不可接受。LinearSelect 提供**确定性的 $O(n)$ 保证**，消灭了性能退化风险。选择 **Q=51~101** 可在保证与性能间取得最佳平衡
3. **Q 越大越好**（在合理范围内）：Q=101 比经典 Q=5 快约 1.9 倍，深度和调用次数均大幅降低
4. **数据已有序时两种算法都快**：受益于 CPU 分支预测和缓存局部性，partition 操作效率大幅提升，与 pivot 质量无关
5. **三路划分有效缓解重复值问题**：重复数据与随机数据耗时相近，因为等值元素被一次性排除

### 7.3 总结

> **QuickSelect 在实际中更快（常数小），LinearSelect 提供了理论上的最坏情况保证但付出了常数代价；增大 Q 可以显著缩小两者差距。**
